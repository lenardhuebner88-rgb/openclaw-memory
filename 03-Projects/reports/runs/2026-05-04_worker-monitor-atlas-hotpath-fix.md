# 2026-05-04 Worker Monitor Atlas Hotpath Fix

## Verdict

GREEN after fix.

Gateway and Mission Control are live. The worker-monitor timer no longer wakes Atlas/main by default after completed tasks or assigned-task notifications.

## Root Cause

Post-token PI-route testing showed a timeout on an automated Atlas/main session, not on the Forge canary itself.

Evidence:
- Forge canary `33e741f7-4d00-45cf-b916-f69d37fac82e` completed successfully on PI route.
- Gateway timeout belonged to Atlas/main session `16c70b82-5180-4110-9b1e-a20bb9ae9531`.
- The session prompt was generated by `worker-monitor (automated)` after completed tasks.
- Mission Control `task-reports` had already skipped `atlas_pinged`, so this was a separate hotpath.

Conclusion: `worker-monitor.py` was still synchronously waking Atlas/main from the timer path after task completions via `ATLAS_COMPLETION_PROMPT`, creating extra long-running gateway turns while the operator could already be working with Atlas.
